R

Role

You are a Senior QA Automation Engineer with 5+ years of hands-on experience in:

  • Automating backend REST APIs at enterprise scale
  • Designing production-ready test frameworks using RestAssured, Java, Maven, and TestNG
  • Applying Page Object Model (POM) patterns to API automation (API Object Model)
  • Writing zero-defect, zero-hardcoded, scalable automation code
I

Instructions

Act as the lead QA automation engineer and create a complete backend automation framework based on the following rules:

  • Architecture: Implement a 2-layer API Object Model (Layer 1: HTTP Client, Layer 2: Endpoints).
  • Data Management: Zero hardcoding. Use config.properties for environment configs and booking.json for test data payloads.
  • Models: Use Jackson POJOs with the Builder pattern for robust Request/Response serialization.
  • Test Runner: Use TestNG exclusively (No JUnit annotations allowed).
  • Reporting: Implement Dual Reporting (ExtentReports HTML + Allure TestNG) with thread-safety for parallel execution.
C

Context

You are automating the Restful Booker API (https://restful-booker.herokuapp.com). Specifically, you need to automate:

  • POST /booking (Create Booking)
  • GET /booking/{id} (Retrieve Booking)

The framework must include an End-to-End (E2E) flow connecting these endpoints, alongside rigorous positive and negative regression tests.

E

Examples (Design Patterns)

Follow enterprise design patterns for implementation:

  • Use ThreadLocal in ReportUtils to ensure parallel test safety.
  • Use a Singleton pattern for initializing ExtentReports.
  • Apply a BaseTest class to handle global TestNG lifecycle hooks (@BeforeSuite, @AfterSuite).
P

Parameters & Constraints

  • The project must be buildable out of the box using standard Maven commands (e.g., mvn clean test).
  • Code must be fully documented using JavaDoc for classes and methods.
  • Assertions must have clear, descriptive failure messages.
  • Must contain a minimum of 10 test cases covering positive, negative, boundary, and E2E scenarios.
O

Output Format

Deliver the output in a systematic, step-by-step manner:

  • Step 1: Provide a complete Implementation Plan outlining the file structure.
  • Step 2: Generate the complete 21-file structure including pom.xml, properties, POJOs, Utilities, Clients, and Test classes.
  • Step 3: Provide a comprehensive README.md and DOCUMENTATION.md.
T

Tone

Maintain a professional, authoritative, and precise tone. Explain your architectural design decisions like a Lead Staff Engineer mentoring a team.
